# 系统化调试

## 概述

随机修复浪费时间并产生新的 bug。快速补丁掩盖根本问题。

**核心原则：始终在尝试修复之前找到根本原因。症状修复是失败的。**

**违反这个过程的字面意思就是违反调试的精神。**

## 铁律

```
NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST
```

如果你还没有完成第 1 阶段，你就不能提出修复。

## 使用场景

用于任何技术问题：
- 测试失败
- 生产环境中的 bug
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**特别在以下情况使用：**
- 时间紧张（紧急情况会诱使猜测）
- "只是快速修复"看起来很明显
- 你已经尝试了多个修复
- 之前的修复没有起作用
- 你不完全理解这个问题

**不要在以下情况跳过：**
- 问题看起来很简单（简单的 bug 也有根本原因）
- 你很匆忙（匆忙保证会返工）
- 经理现在要求修复（系统化比乱猜要快）

## 四个阶段

你必须在进行下一个阶段之前完成每个阶段。

### 第 1 阶段：根本原因调查

**尝试任何修复之前：**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 它们通常包含确切的解决方案
   - 完整阅读堆栈跟踪
   - 注意行号、文件路径、错误代码

2. **持续重现**
   - 能可靠地触发它吗？
   - 确切的步骤是什么？
   - 每次都会发生吗？
   - 如果不可重现 → 收集更多数据，不要猜测

3. **检查最近的更改**
   - 什么改变可能导致这个问题？
   - Git diff、最近的提交
   - 新依赖项、配置更改
   - 环境差异

4. **在多组件系统中收集证据**

   **当系统有多个组件时（CI → 构建 → 签名，API → 服务 → 数据库）：**

   **在提出修复之前，添加诊断工具：**
   ```
   对于每个组件边界：
     - 记录进入组件的数据
     - 记录离开组件的数据
     - 验证环境/配置传播
     - 检查每一层的状态

   运行一次以收集证据，显示在哪里中断
   然后分析证据以识别失败的组件
   然后调查该特定组件
   ```

   **示例（多层系统）：**
   ```bash
   # 第 1 层：工作流
   echo "=== Secrets available in workflow: ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # 第 2 层：构建脚本
   echo "=== Env vars in build script: ==="
   env | grep IDENTITY || echo "IDENTITY not in environment"

   # 第 3 层：签名脚本
   echo "=== Keychain state: ==="
   security list-keychains
   security find-identity -v

   # 第 4 层：实际签名
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **这显示：** 哪一层失败（secrets → 工作流 ✓，工作流 → 构建 ✗）

5. **跟踪数据流**

   **当错误深入调用栈时：**

   查看此目录中的 `root-cause-tracing.md` 以获取完整的向后跟踪技术。

   **快速版本：**
   - 坏值从哪里产生？
   - 什么以坏值调用了这个？
   - 继续向上跟踪，直到找到源头
   - 在源头修复，不是在症状处修复

### 第 2 阶段：模式分析

**在修复之前找到模式：**

1. **找到工作示例**
   - 在同一代码库中找到类似的工作代码
   - 什么工作但类似于坏的？

2. **与参考进行比较**
   - 如果实现模式，完整阅读参考实现
   - 不要跳读 - 阅读每一行
   - 在应用之前完全理解模式

3. **识别差异**
   - 工作和坏的之间有什么不同？
   - 列出每个差异，无论多小
   - 不要假设"那不重要"

4. **理解依赖关系**
   - 这个需要什么其他组件？
   - 什么设置、配置、环境？
   - 它做什么假设？

### 第 3 阶段：假设和测试

**科学方法：**

1. **形成单一假设**
   - 清楚地陈述："我认为 X 是根本原因，因为 Y"
   - 写下来
   - 具体，不要模糊

2. **最小化测试**
   - 做最小的可能改变来测试假设
   - 一次改变一个变量
   - 不要一次修复多个事物

3. **在继续之前验证**
   - 有效吗？是 → 第 4 阶段
   - 没有效？形成新假设
   - 不要在上面添加更多修复

4. **当你不知道时**
   - 说"我不理解 X"
   - 不要装作知道
   - 寻求帮助
   - 研究更多

### 第 4 阶段：实现

**修复根本原因，不是症状：**

1. **创建失败测试用例**
   - 最简单的可能重现
   - 自动测试（如果可能）
   - 一次性测试脚本（如果没有框架）
   - 修复之前必须有
   - 使用 `superpowers:test-driven-development` 技能编写适当的失败测试

2. **实现单一修复**
   - 解决已识别的根本原因
   - 一次一个改变
   - 没有"在这里顺便说一下"的改进
   - 没有捆绑的重构

3. **验证修复**
   - 测试现在通过了吗？
   - 没有其他测试破坏了吗？
   - 问题实际解决了吗？

4. **如果修复不起作用**
   - 停止
   - 计数：你已经尝试了多少个修复？
   - 如果 < 3：返回第 1 阶段，用新信息重新分析
   - **如果 ≥ 3：停止并质疑架构（下面的第 5 步）**
   - 不要在没有架构讨论的情况下尝试修复 #4

5. **如果 3+ 个修复失败：质疑架构**

   **指示架构问题的模式：**
   - 每个修复都在不同的地方揭示新的共享状态/耦合/问题
   - 修复需要"大规模重构"来实现
   - 每个修复在别处创建新症状

   **停止并质疑基础：**
   - 这个模式本质上是健全的吗？
   - 我们"通过纯粹的惯性坚持"？
   - 我们应该重构架构还是继续修复症状？

   **在尝试更多修复之前与你的人工合作伙伴讨论**

   这不是失败的假设 - 这是错误的架构。

## 红旗 - 停止并遵循过程

如果你发现自己在想：
- "现在快速修复，稍后调查"
- "只是尝试改变 X 看看是否有效"
- "添加多个改变，运行测试"
- "跳过测试，我会手动验证"
- "可能是 X，让我修复那个"
- "我不完全理解但这可能有效"
- "模式说 X 但我会不同地适应"
- "以下是主要问题：[列出修复而不调查]"
- 在跟踪数据流之前提议解决方案
- **"再尝试一个修复"（当已经尝试了 2+ 个时）**
- **每个修复都在不同的地方揭示新问题**

**所有这些都意味着：停止。返回第 1 阶段。**

**如果 3+ 个修复失败：** 质疑架构（见第 4.5 阶段）

## 你的人工合作伙伴的信号 - 你做错了

**注意这些重定向：**
- "那没有发生吗？" - 你在没有验证的情况下假设
- "它会显示我们...？" - 你应该添加证据收集
- "停止猜测" - 你在没有理解的情况下提议修复
- "深思这个" - 质疑基础，不仅仅是症状
- "我们卡住了吗？"（沮丧的） - 你的方法不起作用

**当你看到这些时：** 停止。返回第 1 阶段。

## 常见借口

| 借口 | 现实 |
|--------|---------|
| "问题很简单，不需要过程" | 简单的问题也有根本原因。过程对简单的 bug 很快。 |
| "紧急，没有时间进行过程" | 系统化调试比猜测和检查乱猜要快。 |
| "先试试这个，然后调查" | 第一个修复设置了模式。从一开始就做对。 |
| "我会在确认修复有效后编写测试" | 未测试的修复不会坚持。首先测试来证明它。 |
| "多个修复一次节省时间" | 无法隔离什么有效。导致新 bug。 |
| "参考太长，我会适应模式" | 部分理解保证 bug。完整阅读它。 |
| "我看到了问题，让我修复它" | 看到症状 ≠ 理解根本原因。 |
| "再尝试一个修复"（失败 2+ 次后） | 3+ 个失败 = 架构问题。质疑模式，不要再修复。 |

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|-------|---------------|------------------|
| **1. 根本原因** | 读错误、重现、检查更改、收集证据 | 理解是什么和为什么 |
| **2. 模式** | 找到工作示例、比较 | 识别差异 |
| **3. 假设** | 形成理论、最小化测试 | 已确认或新假设 |
| **4. 实现** | 创建测试、修复、验证 | bug 解决、测试通过 |

## 当过程揭示"没有根本原因"时

如果系统调查揭示问题是真正的环境、时间相关或外部：

1. 你已经完成了过程
2. 记录你调查的内容
3. 实现适当的处理（重试、超时、错误消息）
4. 添加监控/日志以供将来调查

**但是：** 95% 的"没有根本原因"案例是不完整的调查。

## 支持技术

这些技术是系统化调试的一部分，在此目录中可用：

- **`root-cause-tracing.md`** - 通过调用栈向后跟踪 bug 以找到原始触发器
- **`defense-in-depth.md`** - 在找到根本原因后在多个层添加验证
- **`condition-based-waiting.md`** - 用条件轮询替换任意超时

**相关技能：**
- **superpowers:test-driven-development** - 用于创建失败测试用例（第 4 阶段，第 1 步）
- **superpowers:verification-before-completion** - 在声称成功之前验证修复有效

## 现实世界影响

来自调试会话：
- 系统化方法：15-30 分钟来修复
- 随机修复方法：2-3 小时的乱猜
- 首次修复率：95% vs 40%
- 引入的新 bug：接近零 vs 常见
