## 审查清单

**代码质量：**
- 关注点是否清晰分离？
- 错误处理是否恰当？
- 类型安全性(如适用)？
- 是否遵循不重复自己(DRY)原则？
- 是否处理边界情况？

**架构：**
- 设计决策是否合理？
- 可伸缩性考虑？
- 性能影响？
- 安全隐患？

**测试：**
- 测试是否真正测试逻辑(而非仅仅模拟)？
- 边界情况是否覆盖？
- 是否需要集成测试？
- 所有测试是否通过？

**需求：**
- 是否满足所有计划需求？
- 实现是否与规范相符？
- 是否有范围蔓延？
- 破坏性变更是否已记录？

**生产就绪性：**
- 迁移策略(如有模式变更)？
- 是否考虑向后兼容性？
- 文档是否完整？
- 是否存在明显错误？

## 关键规则

**要做的：**
- 按实际严重程度分类(不是所有问题都是严重问题)
- 具体说明(文件:行号，而不是含糊其辞)
- 解释问题的重要性
- 认可优势
- 给出明确的判决

**不要做的：**
- 在没有检查的情况下说"看起来不错"
- 将细节问题标记为严重
- 对未审查的代码给出反馈
- 模糊其辞("改进错误处理")
- 避免给出明确的判决
