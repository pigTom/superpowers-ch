# 测试反模式

**加载此参考时机：** 编写或修改测试、添加模拟或想要向生产代码添加测试专用方法时。

## 概述

测试必须验证真实行为，而不是模拟行为。模拟是一种隔离手段，而非被测试的对象。

**核心原则：** 测试代码的实际行为，而非模拟的行为。

**严格遵循测试驱动开发(TDD)可以防止这些反模式。**

## 铁则

```
1. 永远不要测试模拟行为
2. 永远不要向生产类添加测试专用方法
3. 永远不要在不理解依赖关系的情况下使用模拟
```

## 反模式1：测试模拟行为

**违反行为：**
```typescript
// ❌ 错误：测试模拟的存在
test('renders sidebar', () => {
  render(<Page />);
  expect(screen.getByTestId('sidebar-mock')).toBeInTheDocument();
});
```

**为什么这是错误的：**
- 你在验证模拟的工作，而非组件的工作
- 当模拟存在时测试通过，不存在时失败
- 无法了解真实行为

**你的人工合作伙伴的更正：** "我们在测试模拟的行为吗？"

**修复：**
```typescript
// ✅ 正确：测试真实组件或不模拟它
test('renders sidebar', () => {
  render(<Page />);  // 不模拟侧边栏
  expect(screen.getByRole('navigation')).toBeInTheDocument();
});

// 或者如果必须模拟侧边栏以隔离：
// 不要对模拟进行断言——测试 Page 在侧边栏存在时的行为
```

### 门函数

```
在对任何模拟元素进行断言之前：
  问自己：「我在测试真实组件行为，还是仅在测试模拟的存在？」

  如果在测试模拟的存在：
    停止——删除断言或取消模拟组件

  改为测试真实行为
```

## 反模式2：生产代码中的测试专用方法

**违反行为：**
```typescript
// ❌ 错误：destroy() 仅在测试中使用
class Session {
  async destroy() {  // 看起来像生产应用接口！
    await this._workspaceManager?.destroyWorkspace(this.id);
    // ... 清理
  }
}

// 在测试中
afterEach(() => session.destroy());
```

**为什么这是错误的：**
- 生产类被测试专用代码污染
- 如果在生产中意外调用则很危险
- 违反你不会需要它(YAGNI)和关注点分离原则
- 混淆对象生命周期与实体生命周期

**修复：**
```typescript
// ✅ 正确：测试工具处理测试清理
// Session 没有 destroy()——在生产中是无状态的

// 在 test-utils/
export async function cleanupSession(session: Session) {
  const workspace = session.getWorkspaceInfo();
  if (workspace) {
    await workspaceManager.destroyWorkspace(workspace.id);
  }
}

// 在测试中
afterEach(() => cleanupSession(session));
```

### 门函数

```
在向生产类添加任何方法之前：
  问自己：「这仅被测试使用吗？」

  如果是：
    停止——不要添加它
    改为将其放在测试工具中

  问自己：「这个类拥有此资源的生命周期吗？」

  如果否：
    停止——错误的类用于此方法
```

## 反模式3：在不理解的情况下使用模拟

**违反行为：**
```typescript
// ❌ 错误：模拟破坏测试逻辑
test('detects duplicate server', () => {
  // 模拟阻止了测试依赖的配置写入！
  vi.mock('ToolCatalog', () => ({
    discoverAndCacheTools: vi.fn().mockResolvedValue(undefined)
  }));

  await addServer(config);
  await addServer(config);  // 应该抛出——但不会！
});
```

**为什么这是错误的：**
- 模拟的方法有测试依赖的副作用(写入配置)
- 为了「安全」而过度模拟会破坏实际行为
- 测试因错误原因通过或神秘地失败

**修复：**
```typescript
// ✅ 正确：在正确级别进行模拟
test('detects duplicate server', () => {
  // 仅模拟慢速部分，保留测试需要的行为
  vi.mock('MCPServerManager'); // 只模拟慢速服务器启动

  await addServer(config);  // 配置已写入
  await addServer(config);  // 检测到重复 ✓
});
```

### 门函数

```
在模拟任何方法之前：
  停止——先别模拟

  1. 问自己：「真实方法有哪些副作用？」
  2. 问自己：「此测试依赖这些副作用中的任何一个吗？」
  3. 问自己：「我完全理解此测试需要什么吗？」

  如果依赖副作用：
    在更低级别进行模拟(实际的慢速/外部操作)
    或使用保留必要行为的测试替身
    不要模拟测试所依赖的高级方法

  如果不确定测试需要什么：
    先用真实实现运行测试
    观察实际需要发生的事
    然后在正确级别添加最少的模拟

  危险信号：
    - 「我会模拟以安全起见」
    - 「这可能很慢，最好模拟它」
    - 在不理解依赖链的情况下模拟
```

## 反模式4：不完整的模拟

**违反行为：**
```typescript
// ❌ 错误：部分模拟——仅包含你认为需要的字段
const mockResponse = {
  status: 'success',
  data: { userId: '123', name: 'Alice' }
  // 缺少：下游代码使用的元数据
};

// 稍后：当代码访问 response.metadata.requestId 时破裂
```

**为什么这是错误的：**
- **部分模拟隐藏结构假设** ——你仅模拟了你知道的字段
- **下游代码可能依赖你未包含的字段** ——无声故障
- **测试通过但集成失败** ——模拟不完整，真实应用接口完整
- **虚假信心** ——测试无法证明真实行为

**铁则：** 按其在现实中的存在样子模拟完整的数据结构，而不仅仅是你的立即测试使用的字段。

**修复：**
```typescript
// ✅ 正确：镜像真实应用接口的完整性
const mockResponse = {
  status: 'success',
  data: { userId: '123', name: 'Alice' },
  metadata: { requestId: 'req-789', timestamp: 1234567890 }
  // 真实应用接口返回的所有字段
};
```

### 门函数

```
在创建模拟响应之前：
  检查：「真实应用接口响应包含哪些字段？」

  行动：
    1. 检查来自文档/示例的实际应用接口响应
    2. 包含系统可能在下游消费的所有字段
    3. 验证模拟与真实响应模式完全匹配

  关键：
    如果你创建模拟，你必须理解整个结构
    当代码依赖省略字段时，部分模拟无声失败

  如果不确定：包含所有已文档化字段
```

## 反模式5：集成测试作为事后

**违反行为：**
```
✅ 实现完成
❌ 未编写测试
「准备好测试」
```

**为什么这是错误的：**
- 测试是实现的一部分，不是可选的后续
- 测试驱动开发(TDD)会捕获这一点
- 不能声称在没有测试的情况下完成

**修复：**
```
测试驱动开发(TDD)循环：
1. 编写失败的测试
2. 实现以通过
3. 重构
4. 然后声称完成
```

## 当模拟变得过于复杂时

**警告信号：**
- 模拟设置比测试逻辑更长
- 模拟所有内容以使测试通过
- 模拟缺少真实组件拥有的方法
- 当模拟更改时测试破裂

**你的人工合作伙伴的问题：** "我们需要在这里使用模拟吗？"

**考虑：** 带有真实组件的集成测试通常比复杂的模拟更简单

## 测试驱动开发(TDD)防止这些反模式

**为什么测试驱动开发(TDD)有帮助：**
1. **先编写测试** → 强制你思考你实际在测试什么
2. **看它失败** → 确认测试测试真实行为，而非模拟行为
3. **最少实现** → 不会有测试专用方法蔓延
4. **真实依赖** → 你在模拟之前看到测试实际需要什么

**如果你在测试模拟行为，你违反了测试驱动开发(TDD)** ——你在不根据真实代码看测试失败的情况下添加了模拟。

## 快速参考

| 反模式 | 修复 |
|---------|------|
| 对模拟元素进行断言 | 测试真实组件或取消模拟 |
| 生产代码中的测试专用方法 | 移到测试工具 |
| 在不理解的情况下使用模拟 | 先理解依赖，最少模拟 |
| 不完整的模拟 | 完整镜像真实应用接口 |
| 测试作为事后 | 测试驱动开发(TDD)——先测试 |
| 过度复杂的模拟 | 考虑集成测试 |

## 危险信号

- 断言检查 `*-mock` 测试标识符
- 方法仅在测试文件中调用
- 模拟设置超过测试的50%
- 移除模拟时测试失败
- 无法解释为什么需要模拟
- 模拟「以防万一」

## 底线

**模拟是隔离的工具，而非被测试的对象。**

如果测试驱动开发(TDD)揭示你在测试模拟行为，你做错了。

修复：测试真实行为，或质疑为什么你在模拟。
