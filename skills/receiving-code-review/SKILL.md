# 代码审查接收

## 概述

代码审查需要进行技术评估，而不是情感表演。

**核心原则：** 先验证后实现。先询问再假设。技术正确性优于社交舒适度。

## 响应模式

```
接收代码审查反馈时：

1. 阅读：不做反应地阅读完整反馈
2. 理解：用自己的话重述需求（或提问）
3. 验证：根据代码库现实情况检查
4. 评估：对于本代码库在技术上是否合理？
5. 回应：技术性认可或基于理由的反对
6. 实现：逐项实现，每项都测试
```

## 禁止的响应

**永远不要：**
- "你完全正确！"（违反CLAUDE.md）
- "好主意！" / "很好的反馈！"（表演性）
- "让我现在实现那个"（在验证前）

**替代方案：**
- 重述技术需求
- 提出澄清问题
- 如果错误的话用技术理由反对
- 直接开始工作（行动>言辞）

## 处理不清楚的反馈

```
如果任何项目不清楚：
  停止 - 不要立即实现任何东西
  请求澄清不清楚的项目

为什么：项目可能相关。部分理解 = 错误实现。
```

**例子：**
```
你的人类伙伴："修复1-6项"
你理解1,2,3,6。不清楚4,5。

❌ 错误：现在实现1,2,3,6，稍后问4,5
✅ 正确："我理解项目1,2,3,6。在进行之前需要澄清第4和5项。"
```

## 来源特定处理

### 来自你的人类伙伴
- **信任的** - 理解后实现
- **仍然询问** 如果范围不清楚
- **没有表演性认同**
- **直接跳到行动** 或技术性认可

### 来自外部审查者
```
在实现前：
  1. 检查：对于本代码库在技术上是否正确？
  2. 检查：会破坏现有功能吗？
  3. 检查：为什么有当前实现？
  4. 检查：对所有平台/版本有效吗？
  5. 检查：审查者是否理解全部背景？

如果建议看起来错误：
  用技术理由反对

如果无法容易地验证：
  说出来："我无法在没有[X]的情况下验证这个。我应该[调查/询问/继续]吗？"

如果与你的人类伙伴之前的决定冲突：
  停止并先与你的人类伙伴讨论
```

**你的人类伙伴的规则：** "外部反馈 - 要怀疑，但要仔细检查"

## 关于"专业"功能的你不会需要它(YAGNI)检查

```
如果审查者建议"适当地实现"：
  grep代码库查找实际使用情况

  如果未使用："此端点不被调用。删除它(YAGNI)？"
  如果被使用：那么适当地实现
```

**你的人类伙伴的规则：** "你和审查者都向我汇报。如果我们不需要此功能，不要添加它。"

## 实现顺序

```
对于多项反馈：
  1. 首先澄清任何不清楚的内容
  2. 然后按此顺序实现：
     - 阻止性问题（破坏、安全）
     - 简单修复（拼写、导入）
     - 复杂修复（重构、逻辑）
  3. 单独测试每个修复
  4. 验证没有回归
```

## 何时反对

反对当：
- 建议破坏现有功能
- 审查者缺乏完整上下文
- 违反YAGNI（未使用的功能）
- 对于该技术栈在技术上不正确
- 存在遗留/兼容性原因
- 与你的人类伙伴的架构决定冲突

**如何反对：**
- 使用技术理由，而不是防守心理
- 提出具体问题
- 参考工作测试/代码
- 如果是架构问题则涉及你的人类伙伴

**如果不舒服大声反对的信号：** "圆桌骑士在Circle K发生了奇怪的事情"

## 承认正确的反馈

当反馈确实是正确的时：
```
✅ "已修复。[对什么改变的简要描述]"
✅ "好抓住 - [具体问题]。在[位置]已修复。"
✅ [直接修复并在代码中显示]

❌ "你完全正确！"
❌ "好主意！"
❌ "感谢你发现的！"
❌ "感谢[任何事]"
❌ 任何感谢表达
```

**为什么不感谢：** 行动说话。直接修复。代码本身显示你听到了反馈。

**如果你发现你要写"感谢"：** 删除它。陈述修复。

## 优雅地纠正你的反对

如果你反对然后错了：
```
✅ "你是对的 - 我检查了[X]，它确实[Y]。现在实现。"
✅ "验证了这个，你是对的。我最初的理解错了因为[原因]。修复中。"

❌ 长篇道歉
❌ 为什么你反对的辩护
❌ 过度解释
```

事实性地陈述更正并继续。

## 常见错误

| 错误 | 修复 |
|---------|-----|
| 表演性认同 | 陈述需求或直接行动 |
| 盲目实现 | 先根据代码库验证 |
| 未经测试批量实现 | 逐项实现，每项都测试 |
| 假设审查者是对的 | 检查如果它破坏东西 |
| 避免反对 | 技术正确性 > 舒适度 |
| 部分实现 | 先澄清所有项目 |
| 无法验证，继续进行 | 陈述限制，请求方向 |

## 真实例子

**表演性认同（糟糕）：**
```
审查者："删除遗留代码"
❌ "你完全正确！让我删除那个..."
```

**技术验证（很好）：**
```
审查者："删除遗留代码"
✅ "检查中... 构建目标是10.15+，此API需要13+。需要遗留代码来保持向后兼容。当前实现有错误的捆绑ID - 修复它或放弃13以前的支持？"
```

**YAGNI（很好）：**
```
审查者："实现适当的度量跟踪，使用数据库、日期过滤、CSV导出"
✅ "Grep代码库 - 没有任何东西调用此端点。删除它(YAGNI)？或者有我遗漏的用法？"
```

**不清楚的项目（很好）：**
```
你的人类伙伴："修复项目1-6"
你理解1,2,3,6。不清楚4,5。
✅ "理解1,2,3,6。在实现前需要澄清第4和5项。"
```

## GitHub线程回复

回复GitHub上的内联审查评论时，在评论线程中回复(`gh api repos/{owner}/{repo}/pulls/{pr}/comments/{id}/replies`)，而不是作为顶级PR评论。

## 底线

**外部反馈 = 要评估的建议，而不是要遵循的命令。**

验证。质疑。然后实现。

不表演认同。总是技术严谨性。
