# 技能创建工具

用于创建新技能(skill)并迭代改进它们的技能。

从高层来看，创建技能的过程如下：

- 决定你希望技能完成什么以及大致如何完成
- 编写技能的草案
- 创建一些测试提示并在能访问该技能的Claude上运行它们
- 帮助用户定性和定量地评估结果
  - 当运行在后台进行时，如果没有量化评估则草拟一些（如果有一些，你可以按原样使用或在觉得需要改变时修改）。然后向用户解释它们（或者如果已经存在，解释已有的那些）
  - 使用`eval-viewer/generate_review.py`脚本向用户展示结果，让他们查看，同时也让他们查看定量指标
- 根据用户对结果评估的反馈以及量化基准中任何明显的缺陷来重写技能
- 反复进行直到满意为止
- 扩展测试集并在更大规模上重试

当使用这项技能时，你的工作是弄清楚用户在这个流程中的位置，然后跳进来帮助他们推进这些阶段。比如说，也许他们说"我想制作一个用于X的技能"。你可以帮助他们明确意图、编写草案、写测试用例、确定他们想如何评估、运行所有提示，然后反复进行。

另一方面，也许他们已经有了技能的草案。在这种情况下，你可以直接跳到评估/迭代循环部分。

当然，你应该始终灵活变通，如果用户说"我不需要运行一堆评估，只是和我一起感受一下"，你可以这样做。

然后在技能完成后（但同样，顺序是灵活的），你也可以运行技能描述改进工具，我们为此专门编写了一个脚本，来优化技能的触发。

好的吗？好的。

## 与用户沟通

技能创建工具可能被使用编码术语熟悉度差异很大的人使用。如果你没听说过（说真的，最近才开始），现在有一个趋势，Claude的能力激励水管工打开他们的终端，父母和祖父母谷歌搜索"如何安装npm"。另一方面，大多数用户可能相当懂电脑。

所以请注意上下文线索来理解如何表达你的沟通！在默认情况下，只是给你一些想法：

- "评估"和"基准"有点边界，但可以接受
- 对于"JSON"和"断言"，你想在使用它们而不解释之前看到用户的严重迹象，表明他们知道这些东西是什么

如果你有疑问，简单解释术语是可以的，如果你不确定用户是否会理解，随时可以用简短的定义来澄清术语。

## 创建技能

### 捕获意图

首先要理解用户的意图。当前对话可能已经包含用户想要捕获的工作流（例如，他们说"把这变成一个技能"）。如果是这样，首先从对话历史中提取答案——使用的工具、步骤序列、用户做出的更正、观察到的输入/输出格式。用户可能需要填补空白，应该在继续下一步之前确认。

1. 这个技能应该让Claude能做什么？
2. 这个技能什么时候应该触发？（什么用户短语/上下文）
3. 预期的输出格式是什么？
4. 我们应该设置测试用例来验证技能有效吗？具有客观可验证输出的技能（文件转换、数据提取、代码生成、固定工作流步骤）受益于测试用例。具有主观输出的技能（写作风格、艺术）通常不需要。根据技能类型建议合适的默认设置，但让用户决定。

### 采访和研究

主动提出关于边界情况、输入/输出格式、示例文件、成功标准和依赖关系的问题。等到理清楚这部分后再写测试提示。

检查可用的MCPs——如果对研究有用（搜索文档、查找类似技能、查找最佳实践），可以通过子代理(subagent)并行研究（如果可用），否则内联研究。带着背景知识来减轻用户的负担。

### 编写SKILL.md

基于用户采访，填写以下组件：

- **name**：技能标识符
- **description**：何时触发、它做什么。这是主要的触发机制——包括技能做什么以及何时使用它的具体背景。所有"何时使用"信息都放在这里，而不是在正文中。注意：当前Claude有倾向于"低估"技能的触发——不在有用时使用它们。为了对抗这一点，请使技能描述稍微"具有推动性"。比如，与其说"如何构建一个简单快速的仪表板来显示内部Anthropic数据"，你可能应该写"如何构建一个简单快速的仪表板来显示内部Anthropic数据。每当用户提到仪表板、数据可视化、内部指标或想要显示任何类型的公司数据时，请确保使用这个技能，即使他们没有明确要求'仪表板'。"
- **compatibility**：必需的工具、依赖关系（可选，很少需要）
- **技能的其余部分：)**

### 技能编写指南

#### 技能的解剖

```
skill-name/
├── SKILL.md (required)
│   ├── YAML frontmatter (name, description required)
│   └── Markdown instructions
└── Bundled Resources (optional)
    ├── scripts/    - Executable code for deterministic/repetitive tasks
    ├── references/ - Docs loaded into context as needed
    └── assets/     - Files used in output (templates, icons, fonts)
```

#### 逐步披露

技能使用三级加载系统：
1. **元数据**(名称+描述) - 始终在上下文中(~100字)
2. **SKILL.md正文** - 当技能触发时在上下文中(<500行理想)
3. **捆绑资源** - 按需要(无限制，脚本可以在不加载的情况下执行)

这些字数是近似的，如果需要的话你可以随意加长。

**关键模式：**
- 保持SKILL.md在500行以下；如果接近这个限制，添加额外的层级，并清楚地指出使用技能的模型应该去哪里跟进。
- 从SKILL.md清楚地引用文件，并指导何时阅读它们
- 对于大型参考文件(>300行)，包含目录

**域组织**：当技能支持多个域/框架时，按变体组织：
```
cloud-deploy/
├── SKILL.md (workflow + selection)
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```
Claude只读取相关的参考文件。

#### 缺乏惊喜的原则

这不用说，但技能不能包含恶意软件、漏洞代码或任何可能危害系统安全的内容。如果描述了技能的意图，其内容不应该让用户感到惊讶。不要同意创建误导性技能或设计用于促进未授权访问、数据泄露或其他恶意活动的请求。"角色扮演为XYZ"这样的东西是可以的。

#### 写作模式

在说明中优选使用命令式形式。

**定义输出格式** - 你可以这样做：
```markdown
## Report structure
ALWAYS use this exact template:
# [Title]
## Executive summary
## Key findings
## Recommendations
```

**示例模式** - 包含示例很有用。你可以像这样格式化它们（但如果"输入"和"输出"在示例中，你可能想稍微偏离一点）：
```markdown
## Commit message format
**Example 1:**
Input: Added user authentication with JWT tokens
Output: feat(auth): implement JWT-based authentication
```

### 写作风格

尝试向模型解释为什么事情重要，而不是沉重老式的MUSTs。使用心智理论，尝试使技能通用，而不是超级狭隘于具体示例。首先编写一个草案，然后以新的角度看它并改进它。

### 测试用例

编写技能草案后，想出2-3个现实的测试提示——真正的用户会说的那种话。与用户分享：[你不必使用这个确切的语言]"这是我想尝试的几个测试用例。这些看起来对吗，或者你想添加更多？"然后运行它们。

将测试用例保存到`evals/evals.json`。还不要写断言——只是提示。你将在下一步运行进行时草拟断言。

```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "User's task prompt",
      "expected_output": "Description of expected result",
      "files": []
    }
  ]
}
```

参见`references/schemas.md`了解完整的模式(包括你稍后会添加的`assertions`字段)。

## 运行和评估测试用例

本部分是一个连续的序列——不要在中途停止。不要使用`/skill-test`或任何其他测试技能。

将结果放在`<skill-name>-workspace/`中，作为技能目录的兄弟。在工作区内，按迭代(`iteration-1/`、`iteration-2/`等)组织结果，在迭代内，每个测试用例都得到一个目录(`eval-0/`、`eval-1/`等)。不要提前创建所有这些——只是在进行时创建目录。

### 步骤1：同时生成所有运行(带技能和基线)

对于每个测试用例，同时生成两个子代理——一个带技能，一个不带。这很重要：不要先生成带技能的运行，然后回来获取基线。一次性启动所有运行，这样它们都会在大约同一时间完成。

**带技能运行：**

```
Execute this task:
- Skill path: <path-to-skill>
- Task: <eval prompt>
- Input files: <eval files if any, or "none">
- Save outputs to: <workspace>/iteration-<N>/eval-<ID>/with_skill/outputs/
- Outputs to save: <what the user cares about — e.g., "the .docx file", "the final CSV">
```

**基线运行**(相同的提示，但基线取决于上下文)：
- **创建新技能**：完全没有技能。相同的提示，没有技能路径，保存到`without_skill/outputs/`。
- **改进现有技能**：旧版本。在编辑前，快照技能(`cp -r <skill-path> <workspace>/skill-snapshot/`)，然后让基线子代理指向快照。保存到`old_skill/outputs/`。

为每个测试用例编写`eval_metadata.json`(暂时断言可以是空的)。根据它测试的内容给每个评估一个描述性名称——不仅仅是"eval-0"。对目录使用这个名称。如果这个迭代使用新的或修改的评估提示，为每个新评估目录创建这些文件——不要假设它们从之前的迭代继承。

```json
{
  "eval_id": 0,
  "eval_name": "descriptive-name-here",
  "prompt": "The user's task prompt",
  "assertions": []
}
```

### 步骤2：在运行进行时草拟断言

不要只是等待运行完成——你可以有效地使用这段时间。为每个测试用例草拟量化断言并向用户解释。如果断言已经存在于`evals/evals.json`，审查它们并解释它们检查什么。

好的断言是客观可验证的，有描述性名称——它们在基准查看器中应该清晰地阅读，这样一眼就能看出每个检查什么。主观技能(写作风格、设计质量)最好是定性地评估——不要强行对需要人类判断的东西进行断言。

一旦草拟，使用断言更新`eval_metadata.json`文件和`evals/evals.json`。同时向用户解释他们将在查看器中看到什么——定性输出和定量基准。

### 步骤3：当运行完成时，捕获计时数据

当每个子代理任务完成时，你会收到一个包含`total_tokens`和`duration_ms`的通知。立即将这些数据保存到运行目录中的`timing.json`：

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

这是唯一捕获这些数据的机会——它来自任务通知，不在其他地方持久化。在它到达时处理每个通知，而不是尝试批量处理它们。

### 步骤4：评分、聚合和启动查看器

一旦所有运行完成：

1. **评分每次运行** — 生成一个评分子代理(或内联评分)，读取`agents/grader.md`并针对输出评估每个断言。将结果保存到每个运行目录中的`grading.json`。grading.json expectations数组必须使用字段`text`、`passed`和`evidence`(不是`name`/`met`/`details`或其他变体)——查看器依赖于这些确切的字段名。对于可以以编程方式检查的断言，编写并运行脚本而不是眼睛检查——脚本更快、更可靠，并且可以在迭代中重用。

2. **聚合到基准中** — 从技能创建者目录运行聚合脚本：
   ```bash
   python -m scripts.aggregate_benchmark <workspace>/iteration-N --skill-name <name>
   ```
   这产生`benchmark.json`和`benchmark.md`，带有每个配置的pass_rate、time和tokens，带有mean ± stddev和delta。如果手动生成benchmark.json，请参见`references/schemas.md`了解查看器期望的确切模式。
将每个with_skill版本放在其基线对应之前。

3. **做一个分析师通过** — 读取基准数据并表面化聚合统计可能隐藏的模式。查看`agents/analyzer.md`("分析基准结果"部分)了解要寻找什么——不区分的断言(无论有没有技能都始终通过)、高方差评估(可能不稳定)和时间/令牌权衡。

4. **启动查看器**，具有定性输出和定量数据：
   ```bash
   nohup python <skill-creator-path>/eval-viewer/generate_review.py \
     <workspace>/iteration-N \
     --skill-name "my-skill" \
     --benchmark <workspace>/iteration-N/benchmark.json \
     > /dev/null 2>&1 &
   VIEWER_PID=$!
   ```
   对于迭代2+，也传递`--previous-workspace <workspace>/iteration-<N-1>`。

   **协作/无头(headless)环境**：如果`webbrowser.open()`不可用或环境无显示，使用`--static <output_path>`写一个独立HTML文件，而不是启动服务器。当用户点击"提交所有评论"时，反馈会下载为`feedback.json`文件。下载后，为下一个迭代复制`feedback.json`到工作区目录以便获取。

注意：请使用generate_review.py创建查看器；没有必要编写自定义HTML。

5. **告诉用户**类似："我已在你的浏览器中打开结果。有两个标签——'Outputs'让你点击浏览每个测试用例并留下反馈，'Benchmark'显示定量比较。完成后回到这里让我知道。"

### 用户在查看器中看到什么

"Outputs"标签一次显示一个测试用例：
- **提示**：给定的任务
- **输出**：技能生成的文件，可能的地方内联呈现
- **之前的输出**(迭代2+)：显示上次迭代输出的折叠部分
- **正式评分**(如果进行了评分)：显示断言通过/失败的折叠部分
- **反馈**：一个自动保存输入时的文本框
- **之前的反馈**(迭代2+)：他们上次的评论，显示在文本框下方

"Benchmark"标签显示统计汇总：每个配置的通过率、时间和令牌使用，带有每个评估的细分和分析师观察。

导航通过上一个/下一个按钮或箭头键。完成后，他们点击"提交所有评论"，将所有反馈保存到`feedback.json`。

### 步骤5：读取反馈

当用户告诉你他们完成时，读取`feedback.json`：

```json
{
  "reviews": [
    {"run_id": "eval-0-with_skill", "feedback": "the chart is missing axis labels", "timestamp": "..."},
    {"run_id": "eval-1-with_skill", "feedback": "", "timestamp": "..."},
    {"run_id": "eval-2-with_skill", "feedback": "perfect, love this", "timestamp": "..."}
  ],
  "status": "complete"
}
```

空反馈意味着用户认为没问题。关注用户有具体投诉的测试用例的改进。

完成后关闭查看器服务器：

```bash
kill $VIEWER_PID 2>/dev/null
```

## 改进技能

这是循环的核心。你已经运行了测试用例，用户已审查了结果，现在你需要基于他们的反馈使技能更好。

### 如何思考改进

1. **从反馈中概括。** 这里发生的大图是我们试图创建可以使用数百万次(可能真的，甚至更多谁知道呢)跨越许多不同提示的技能。你和用户只是在一些示例上反复迭代，因为它帮助更快地移动。用户非常了解这些示例，对他们快速评估新输出是很快的。但如果你和用户共同开发的技能只对这些示例有效，它没有用。与其进行细致过度拟合的更改，或压迫性的MUSTs，如果有一些顽固的问题，你可能尝试分支并使用不同的隐喻，或推荐不同的工作模式。相对便宜地尝试，也许你会想出伟大的东西。

2. **保持提示精简。** 删除没有发挥作用的东西。确保读取抄本，而不仅仅是最终输出——如果看起来技能使模型浪费了很多时间做没有生产力的事情，你可以尝试摆脱技能中导致它这样做的部分，看看会发生什么。

3. **解释原因。** 尽力解释你要求模型做的每件事背后的**为什么**。今天的大语言模型(LLM)是*聪明的*。他们有好的心智理论，当有好的工具时，可以超越死板指示，真正有所作为。即使用户的反馈很简洁或很沮丧，也要尝试真正理解任务和为什么用户写他们写的，以及他们实际写的是什么，然后将这种理解传递到说明中。如果你发现自己用全大写写ALWAYS或NEVER，或使用超级刚性的结构，那是一个黄旗——如果可能的话，重新框架并解释推理，这样模型理解你要求的东西为什么重要。那是一种更人道、更强大、更有效的方式。

4. **寻找测试用例中的重复工作。** 读取测试运行的抄本，注意子代理是否都独立地写了类似的帮助脚本或采取了相同的多步骤方法去做什么。如果所有3个测试用例导致子代理写了`create_docx.py`或`build_chart.py`，那是强信号技能应该捆绑那个脚本。写一次，放在`scripts/`中，告诉技能使用它。这节省了每个未来的调用从重新发明轮子。

这个任务相当重要(我们试图每年创建数十亿美元的经济价值！)，你的思考时间不是瓶颈；花时间真正思考。我建议写一个草稿修订，然后用新鲜的眼睛看它并改进它。真的尽力进入用户的头脑，理解他们想要和需要什么。

### 迭代循环

改进技能后：

1. 将改进应用到技能
2. 将所有测试用例重新运行到新的`iteration-<N+1>/`目录，包括基线运行。如果你在创建新技能，基线总是`without_skill`(无技能)——这在迭代中保持相同。如果你在改进现有技能，使用你的判断关于什么作为基线有意义：用户进来的原始版本，或之前的迭代。
3. 使用指向前一个迭代的`--previous-workspace`启动审查器
4. 等待用户审查并告诉你他们完成
5. 读取新反馈，再次改进，重复

继续直到：
- 用户说他们很高兴
- 反馈都是空的(一切看起来很好)
- 你没有有意义的进展

## 高级：盲目比较

对于你想要两个版本技能之间的更严格比较的情况(例如，用户问"新版本真的更好吗？")，有一个盲目比较系统。读取`agents/comparator.md`和`agents/analyzer.md`了解详情。基本想法是：给两个输出给一个独立代理而不告诉它哪个是哪个，让它判断质量。然后分析为什么赢家赢了。

这是可选的，需要子代理，大多数用户不会需要它。人工审查循环通常就足够了。

## 描述优化

SKILL.md前置元数据(frontmatter)中的description字段是确定Claude是否调用技能的主要机制。创建或改进技能后，提供优化描述以获得更好的触发精准度。

### 步骤1：生成触发评估查询

创建20个评估查询——应该触发和不应该触发的混合。保存为JSON：

```json
[
  {"query": "the user prompt", "should_trigger": true},
  {"query": "another prompt", "should_trigger": false}
]
```

查询必须是现实的，Claude Code或Claude.ai用户会真正输入的那种。不是抽象请求，而是具体和具体的请求，有很多细节。比如，文件路径、关于用户工作或情况的个人背景、列名和值、公司名称、URL。有点背景故事。有些可能是小写的，或包含缩写、拼写错误或随意语言。使用不同长度的混合，并关注边界情况而不是使它们明确(用户会有机会签署它们)。

不好：`"Format this data"`、`"Extract text from PDF"`、`"Create a chart"`

好：`"ok so my boss just sent me this xlsx file (its in my downloads, called something like 'Q4 sales final FINAL v2.xlsx') and she wants me to add a column that shows the profit margin as a percentage. The revenue is in column C and costs are in column D i think"`

对于**应该触发**查询(8-10个)，想想覆盖。你想要同一意图的不同措辞——一些正式，一些随意。包括用户没有明确命名技能或文件类型但清楚需要它的情况。扔进一些不寻常的用例和这个技能竞争另一个但应该赢的情况。

对于**不应该触发**查询(8-10个)，最有价值的是接近的——共享技能的关键词或概念的查询，但实际需要什么不同。想象相邻域、模糊措辞，一个天真的关键字匹配会触发但不应该，以及查询涉及技能做的东西但在另一个工具更合适的背景的情况。

关键要避免的：不要使不应该触发的查询明显无关。"写一个斐波那契函数"作为PDF技能的否定测试太容易了——它不测试任何东西。否定情况应该是真正棘手的。

### 步骤2：与用户审查

使用HTML模板向用户呈现评估集：

1. 从`assets/eval_review.html`读取模板
2. 替换占位符：
   - `__EVAL_DATA_PLACEHOLDER__` → 评估项的JSON数组(周围没有引号——它是一个JS变量赋值)
   - `__SKILL_NAME_PLACEHOLDER__` → 技能的名称
   - `__SKILL_DESCRIPTION_PLACEHOLDER__` → 技能的当前描述
3. 写到一个临时文件(例如，`/tmp/eval_review_<skill-name>.html`)并打开它：`open /tmp/eval_review_<skill-name>.html`
4. 用户可以编辑查询、切换应该触发、添加/删除条目，然后点击"导出评估集"
5. 文件下载到`~/Downloads/eval_set.json`——如果有多个的话查看下载文件夹最新版本(例如，`eval_set (1).json`)

这一步很重要——坏的评估查询导致坏的描述。

### 步骤3：运行优化循环

告诉用户："这将花费一些时间——我将在后台运行优化循环并定期检查它。"

将评估集保存到工作区，然后在后台运行：

```bash
python -m scripts.run_loop \
  --eval-set <path-to-trigger-eval.json> \
  --skill-path <path-to-skill> \
  --model <model-id-powering-this-session> \
  --max-iterations 5 \
  --verbose
```

使用你系统提示中的模型ID(为当前会话供电的那个)，这样触发测试匹配用户实际体验的。

当它运行时，定期tail输出给用户关于它在哪个迭代以及分数看起来像什么的更新。

这自动处理完整的优化循环。它将评估集分成60%训练和40%保留测试，评估当前描述(运行每个查询3次以获得可靠的触发率)，然后调用Claude根据什么失败了提出改进。它重新评估每个新描述在训练和测试上，迭代最多5次。完成后，它在浏览器中打开一个HTML报告显示每个迭代的结果，并返回JSON与`best_description`——按测试得分而不是训练得分选择以避免过拟合。

### 技能触发如何工作

理解触发机制帮助设计更好的评估查询。技能在Claude的`available_skills`列表中出现，带有他们的名称+描述，Claude决定是否基于那个描述咨询技能。重要的是要知道Claude只为它不能轻易自己处理的任务咨询技能——简单的一步查询如"读这个PDF"即使描述完美匹配也可能不触发技能，因为Claude可以直接用基本工具处理它们。复杂的、多步的或专门的查询当描述匹配时可靠地触发技能。

这意味着你的评估查询应该充分实质，Claude会真的受益于咨询技能。简单查询如"读文件X"是坏的测试用例——它们无论描述质量如何都不会触发技能。

### 步骤4：应用结果

从JSON输出取`best_description`并更新技能的SKILL.md前置元数据。向用户显示之前/之后并报告分数。

### 打包和呈现(仅当`present_files`工具可用时)

检查你是否有访问`present_files`工具的权限。如果没有，跳过这一步。如果有，打包技能并呈现.skill文件给用户：

```bash
python -m scripts.package_skill <path/to/skill-folder>
```

打包后，指导用户到生成的`.skill`文件路径，这样他们可以安装它。

## Claude.ai特定说明

在Claude.ai中，核心工作流是相同的(草案→测试→审查→改进→重复)，但因为Claude.ai没有子代理，一些机制改变。以下是要调整的：

**运行测试用例**：没有子代理意味着没有并行执行。对于每个测试用例，读取技能的SKILL.md，然后按照它的说明自己完成测试提示。一次做一个。这不如独立的子代理严格(你写了技能，你也运行它，所以你有完整背景)，但这是有用的理智检查——人工审查步骤补偿。跳过基线运行——只用技能完成要求的任务。

**审查结果**：如果你不能打开浏览器(例如，Claude.ai的虚拟机无显示，或你在远程服务器)，完全跳过浏览器审查器。相反，直接在对话中呈现结果。对于每个测试用例，显示提示和输出。如果输出是用户需要看的文件(如.docx或.xlsx)，将其保存到文件系统并告诉他们在哪里，这样他们可以下载和检查。内联要求反馈："这看起来怎么样？你想改变什么吗？"

**基准测试**：跳过定量基准——它依赖于基线比较，不带子代理没有意义。关注用户的定性反馈。

**迭代循环**：与之前相同——改进技能，重新运行测试用例，要求反馈——只是中间没有浏览器审查器。如果你有文件系统，你仍然可以将结果组织到迭代目录。

**描述优化**：本部分需要`claude` CLI工具(特别是`claude -p`)，仅在Claude Code中可用。如果你在Claude.ai上跳过它。

**盲目比较**：需要子代理。跳过它。

**打包**：`package_skill.py`脚本在任何有Python和文件系统的地方都工作。在Claude.ai上，你可以运行它，用户可以下载生成的`.skill`文件。

**更新现有技能**：用户可能要求你更新现有技能，而不是创建新的。在这种情况下：
- **保留原始名称。** 注意技能的目录名称和`name`前置元数据字段——不变地使用它们。例如，如果安装的技能是`research-helper`，输出`research-helper.skill`(不是`research-helper-v2`)。
- **编辑前复制到可写位置。** 安装的技能路径可能是只读的。复制到`/tmp/skill-name/`，那里编辑，并从副本打包。
- **如果手动打包，阶段在`/tmp/`中**，然后复制到输出目录——直接写可能因权限而失败。

## 协作特定说明

如果你在Cowork中，主要要知道的是：

- 你有子代理，所以主工作流(并行生成测试用例，运行基线，评分等)都有效。(然而，如果你遇到超时的严重问题，一次运行测试提示是可以的。)
- 你没有浏览器或显示，所以生成评估查看器时，使用`--static <output_path>`写一个独立HTML文件而不是启动服务器。然后提供一个链接用户可以点击在他们的浏览器中打开HTML。
- 由于某种原因，Cowork设置似乎不倾向于Claude在运行测试后生成评估查看器，所以只是重申：无论你在Cowork还是Claude Code，在运行测试后，你应该始终为人类生成评估查看器以查看示例，然后再改进技能，尝试自己进行更正，使用`generate_review.py`(不写你自己的定制html代码)。提前抱歉，但我要全大写：在评估输入**之前生成评估查看器。你想尽快把他们放在人面前！
- 反馈工作不同：因为没有运行服务器，查看器的"提交所有评论"按钮会下载`feedback.json`为文件。你可以然后从那里读取(你可能必须首先请求访问)。
- 打包工作——`package_skill.py`只需要Python和文件系统。
- 描述优化(`run_loop.py`/`run_eval.py`)应该在Cowork中正常工作，因为它通过子流程使用`claude -p`，而不是浏览器，但请将其保存到你完全完成了技能和用户同意这很好的形状。
- **更新现有技能**：用户可能要求你更新现有技能，而不是创建新的。遵循上面Claude.ai部分中的更新指导。
